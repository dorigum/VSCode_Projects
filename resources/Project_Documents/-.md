  1단계: 원칙 - "구체적인 설정이 일반적인 설정보다 우선한다" (Priority Rule)


  데이터베이스 설계에서 가장 널리 쓰이는 '계층적 오버라이딩(Hierarchical Overriding)' 원리입니다.


   * 일반적 설정 (Category Level): "이 카테고리에 속한 모든 메뉴는 기본적으로 A, B 옵션을 가진다." (넓은 범위)
   * 구체적 설정 (Menu Level): "하지만 이 특정 메뉴(청포도에이드)는 C, D 옵션을 가진다." (좁고 구체적인 범위)


  논리: 시스템은 항상 '가장 구체적인 정보'를 먼저 찾고, 그 정보가 없을 때만 '일반적인 정보'로 넘어갑니다.

  ---

  2단계: 작동 메커니즘 - "존재 여부 확인 (Check & Fallback)"

  실제 코드가 실행되는 내부 로직은 다음과 같은 '조건부 조회' 순서를 따릅니다.


   1. [1순위 확인] 메뉴별 특화 정보 (MENU_OPTION_GROUP):
       * 먼저 MENU_OPTION_GROUP 테이블에서 해당 메뉴(ID)에 연결된 옵션이 있는지 조회합니다.
       * 결과가 있다면? "이 메뉴는 전용 설정이 있구나!"라고 판단하고, 여기서 가져온 옵션들만 화면에 뿌립니다. (이때 카테고리  
         기본값은 아예 무시됩니다.)
   2. [2순위 확인] 카테고리 기본 정보 (CATEGORY_OPTION_GROUP):
       * 만약 1순위 조회 결과가 비어 있다면(Null/Empty)? "이 메뉴는 특별한 설정이 없는 일반 메뉴구나!"라고 판단합니다.        
       * 그제서야 CATEGORY_OPTION_GROUP으로 가서 해당 카테고리의 기본 옵션들을 가져옵니다.

  이것이 바로 '예외 처리'의 원리입니다. 메뉴별 테이블에 데이터를 넣는 행위 자체가 "기본값을 쓰지 않겠다"는 선언이 되는 것이죠.

  ---


  3단계: 실례 - '청포도에이드'의 경우


   * 상황: 논커피 카테고리 기본값은 [사이즈, 온도]인데, 에이드는 [사이즈, 탄산]으로 바꾸고 싶음.
   * 실행: MENU_OPTION_GROUP에 청포도에이드 전용으로 [사이즈, 탄산] 데이터 2줄을 넣습니다.
   * 로직의 흐름:
       1. 시스템이 청포도에이드의 전용 옵션을 찾음 -> [사이즈, 탄산] 발견!
       2. 발견했으므로, 카테고리 기본값인 [사이즈, 온도]는 조회조차 하지 않음.
       3. 화면에는 [사이즈, 탄산]만 노출됨. (결과적으로 '온도'가 삭제된 것처럼 보임)

  ---

  💡 이 방식이 '삭제'보다 논리적으로 우월한 이유

  팀원이 "그냥 메뉴 테이블에 다 넣고 관리하면 안 돼?"라고 물을 때의 반박 논리입니다.


   1. 템플릿(Template) 효과: CATEGORY_OPTION_GROUP은 일종의 '마스터 템플릿'입니다. 템플릿이 있으면 90%의 일반 메뉴는 설정할
      필요가 없습니다. 템플릿을 없애면 100% 모든 메뉴를 수동 설정해야 합니다.
   2. 유지보수의 정석: 만약 모든 메뉴에 일일이 옵션을 다 넣었는데, 나중에 공통 옵션 하나가 바뀌면 수백 개의 메뉴 데이터를  
      수정해야 합니다. 하지만 사용자님의 방식은 카테고리 기본값(템플릿) 하나만 수정하면 예외 설정이 없는 모든 메뉴에 즉시  
      적용됩니다.

  요약:
  사용자님의 설계는 "기본적으로는 자동(Category Default), 필요할 때만 수동(Menu Override)" 시스템입니다.


  팀원이 제안하는 방식은 "처음부터 끝까지 100% 수동" 시스템입니다. 어떤 게 더 똑똑하고 장애가 적은 설계인지는 명확합니다.
